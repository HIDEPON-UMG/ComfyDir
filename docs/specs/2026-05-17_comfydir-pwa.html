ComfyDir PWA 化

現行 Web 版に PWA (manifest + Service Worker + タスクトレイ launcher) を載せる。ネイティブ exe アプリ化案は撤回し、最小コストで「ネイティブアプリ感」「タスクトレイ常駐」「インストール導線」「オフラインフォールバック」を獲得する方針。

draft v0.9.0-web + PWA plan 承認済

背景

前段 — PySide6 ネイティブ exe 案は却下

2026-05-17、PySide6 全面リライト案 (3 週間相当の計画) の Phase 2 でタイル描画破綻 (QListView IconMode + 自然アスペクト sizeHint の整合不可) が露呈し、却下された。HEAD は v0.9.0-web タグに巻き戻し済 (PySide6 作業は stash@{0} に温存)。

代わりに「現行 Web 版 + PWA」で同じ UX 価値 (ネイティブ感 / 高速起動 / オフライン) を最小コストで取りに行く。

採用しなかった案

PWA 化のアプローチを 3 つ比較し、当初「pystray + Edge --app=」 を採用したが、--app= モードでは Chrome が display-mode: standalone と判定し beforeinstallprompt が発火しない仕様が判明したため、「pystray + Chrome --new-window + 専用 user-data-dir」 へ最終切替。決め手は「依存最小 (pystray のみ)」「PWA インストール導線が機能する」「Profile Badging を抑止して独自アイコンが綺麗に出る」。

pystray + Chrome --new-window + 専用プロファイル

採用

タスクトレイに常駐、クリックで chrome --user-data-dir=...\ComfyDir\ChromeProfile --new-window http://127.0.0.1:8765 --window-size=1440,900 で通常ウィンドウ起動。

PROS
  • 依存は pystray のみ (pywin32 不要)
  • 通常ウィンドウなので beforeinstallprompt が発火 → 画面内インストールボタンが機能
  • 専用プロファイル (Google 未ログイン) → Profile Badging のアバターオーバーレイなし
  • 普段使い Chrome と完全分離 (履歴・cookie・拡張機能)
  • SQLite 含む既存資産は無修正
CONS
  • 初回起動時に専用プロファイルディレクトリ作成のため Chrome 起動が少し遅い
  • 「アプリ感」(タスクバー独自アイコン + standalone) を得るには PWA インストール必須 (二段構え運用)

pywebview / Eel

却下

Python から WebView2 / CEF を呼んでブラウザウィンドウをホスト。

PROS

  • JS bridge でファイル D&D も拡張可

  • 本目的「PWA 化」の本筋を外れる
  • Edge シェルの D&D 互換性が壊れる可能性

# launcher.py の核 — Win32 API でウィンドウ前面化
import ctypes, shutil, subprocess
_user32 = ctypes.windll.user32
_user32.FindWindowW.restype = ctypes.c_void_p

def open_or_focus_window():
    hwnd = _user32.FindWindowW(None, "ComfyDir")
    if hwnd:
        _user32.ShowWindow(hwnd, 9)  # SW_RESTORE
        _user32.SetForegroundWindow(hwnd)
        return
    browser = shutil.which("chrome") or shutil.which("msedge")
    CHROME_PROFILE_DIR.mkdir(parents=True, exist_ok=True)
    subprocess.Popen(
        [browser, f"--user-data-dir={CHROME_PROFILE_DIR}",
         "--new-window", URL, "--no-first-run",
         "--no-default-browser-check", "--window-size=1440,900"],
        creationflags=0x08000000 | 0x00000008,
    )
// sw.js — fetch ハンドラのルーティング骨格
self.addEventListener("fetch", (e) => {
  const req = e.request, url = new URL(req.url);
  if (req.method !== "GET") return;
  if (url.origin !== self.location.origin) return;
  if (/^\/api\/images\/\d+\/thumb/.test(url.pathname))
    return e.respondWith(cacheFirst(req, RUNTIME));
  if (url.pathname === "/api/prompt-category-map")
    return e.respondWith(cacheFirst(req, RUNTIME));
  if (url.pathname.startsWith("/api/"))
    return e.respondWith(networkFirst(req, RUNTIME));
  if (req.mode === "navigate")
    return e.respondWith(
      networkFirst(req, RUNTIME).catch(() => caches.match("/offline.html"))
    );
  e.respondWith(cacheFirst(req, RUNTIME));
});

リスクと対応

#リスク対応
R1pystray が VBS hidden 起動下でコンソール窓を一瞬出すsubprocess.Popen で CREATE_NO_WINDOW \| DETACHED_PROCESS 明示。pythonw.exe で launcher 起動
R2PWA インストール後の app-id 起動と --app= の挙動差README で「インストール後はスタートメニュー優先、--app= 直起動は tray 経由のみ」と明記
R3SW キャッシュバージョン管理ミスで古い app.js が残る(a) 登録時 updateViaCache: 'none' (b) /sw.js に Cache-Control: no-cache (c) VERSION 定数を変えれば古い cache を自動破棄
R4localhost で SW が動くかlocalhost は secure context として SW 有効 (W3C 仕様)。smoke 2 で検証
R5既存 .lnk と PWA install 生成 shortcut の二重管理README で「PWA インストールは任意」と明記
R6FindWindowW 完全一致で末尾 - Microsoft Edge が付いて外すフォールバック: EnumWindows で部分一致。smoke 4 で対応
R7uvicorn を thread で起動するとシグナルハンドラ警告uvicorn.Server.install_signal_handlers = lambda *a, **k: None でパッチ
R8start.vbs 文字コード破壊ASCII 維持、新規 MsgBox 文言も英語

パラメータ

SW のキャッシュバージョンを調整して、古い app.js を強制的に破棄したいときに「次回 SW activate でキャッシュ全消し」を起こせる。スライダで値を選んで「プロンプトとしてコピー」で Claude に戻す。

v1

指針
• v1: 初回リリース。precache 8 件 + runtime cache
• v2〜: app.js / style.css / index.html 改修で挙動が変わったとき。activate で旧 cache (comfydir-precache-v(N-1) 等) を caches.delete で掃除
• バージョン上げ忘れると古い app.js が browser cache から返って改修が反映されない (R3 の本丸)

補足
sw.js 冒頭の const VERSION = "vN" を直接編集。VERSION 単体を変えれば PRECACHE と RUNTIME のキー名が自動的に新しくなり、旧 cache は activate で消える。
